# 智慧居家總店選址評估系統說明手冊

版本：v1.1  
更新日期：2026-06-23  
適用對象：前端工程師、全端工程師、產品經理、UI/UX 設計師、房仲專家、展店顧問  
系統型態：純前端 Web App，無後端、無資料庫、無付費地圖 API 即可運作

## 1. 系統目的

本系統用於協助智慧居家、展示中心、總店或旗艦店進行候選店面比較。它把選址決策中常見的主觀評分、客觀條件、財務坪效、交通距離、硬性門檻與專家出價分析整合為一套可重複使用的量化流程。

系統核心目標：

1. 同時比較多個候選店面。
2. 將房仲或展店顧問提供的物件資訊結構化。
3. 讓決策者調整策略權重，例如平衡模式、品牌優先、投報優先。
4. 讓使用者統一設定全域距離目標，例如住家、國小、百貨公司、高鐵站、捷運站。
5. 讓使用者設定全域硬性門檻，例如預算上限、最低建坪、最低面寬。
6. 即時計算加權總分、財務坪效、市場估值價、建議下錨價、最高追價。
7. 在評比排行頁以轉置表格呈現所有物件差異。
8. 評比排行項目列可由使用者拖拉排序，方便把會議最關心的價格資訊集中比較。
9. 以單一 CSV 保存完整評估案，並可再次匯入繼續評估。
10. 額外支援評比排行對比表 CSV 匯出，供決策會議使用。

| 類別 | 建議 |

## 3. 核心資料模型

系統主要維護五類資料：

1. `weights`：策略評分權重。
2. `properties`：候選物件資料陣列。
3. `distanceTargets`：全域距離目標。
4. `constraints`：全域期待指標與硬性門檻。
5. UI state：頁面切換、Accordion、刪除 Modal、是否隱藏不符條件物件等。

### 3.1 權重資料 `weights`

權重使用 key-value 物件保存。每個值建議為 0 到 50 的整數。

```js
{
  loc1: 15,
  loc2: 10,
  fac1: 10,
  fac2: 10,
  spc1: 10,
  spc2: 15,
  fin1: 10,
  fin2: 10,
  hd1: 10
}
```

| key | 名稱 | 群組 | 用途 |
| --- | --- | --- | --- |
| `loc1` | 商圈定位 | 區位 | 商圈、地段、品牌曝光與客群契合度 |
| `loc2` | 接駁停車 | 區位 | 停車、捷運、高鐵、交通便利性 |
| `fac1` | 能見度 | 店況 | 門面、招牌、車流與人流可見性 |
| `fac2` | 格局方正 | 店況 | 展示、動線、施工便利性 |
| `spc1` | 挑高條件 | 空間 | 夾層、展示牆、體驗空間 |
| `spc2` | 坪數採光 | 空間 | 使用坪效、採光與開放感 |
| `fin1` | 總價預算 | 財務 | 是否符合購置預算 |
| `fin2` | 單價急售 | 財務 | 單價、議價空間、急售機會 |
| `hd1` | 既有硬體 | 硬體 | 空調、水電、消防、裝修可沿用程度 |

### 3.2 全域距離目標 `distanceTargets`

距離目標由使用者設定一次，所有物件共用。

```js
[
  { id: "target1", name: "高鐵站", address: "高鐵桃園站" },
  { id: "target2", name: "捷運站", address: "桃園捷運A19站" },
  { id: "target3", name: "停車場", address: "附近停車場" },
  { id: "target4", name: "自訂基準點", address: "" }
]
```

設計理由：

1. 住家、學校、百貨公司、指定客戶公司等目標會因決策目的而變。
2. 這些資料屬於評估案全域條件，不應混入單一物件資料。
3. CSV 匯出時獨立成 `GLOBAL_DISTANCE_TARGETS` 區段。

目前建議最多 5 個距離目標。

### 3.3 全域期待指標 `constraints`

`constraints` 是一次設定、套用到所有物件的硬性門檻與估價參數。

```js
{
  targetMaxPrice: 0,
  targetMinArea: 0,
  targetMinWidth: 0,
  renovationDeductionPerPing: 5
}
```

| key | 名稱 | 單位 | 說明 |
| --- | --- | --- | --- |
| `targetMaxPrice` | 預算上限 | 萬 | 若物件總價超過此值，標示 `預算超標` |
| `targetMinArea` | 最低建坪 | 坪 | 若建坪低於此值，標示 `坪數不足` |
| `targetMinWidth` | 最低面寬 | 米 | 若面寬低於此值，標示 `面寬不足` |
| `renovationDeductionPerPing` | 硬體扣除額 | 萬/坪 | 當硬體評分低於 5 分時，用於建議買價扣除 |

當門檻值為 0 時，代表不啟用該門檻。

系統提供「隱藏不符條件物件」切換：

1. 關閉時：所有物件仍顯示，但不符者顯示紅色標籤。
2. 開啟時：不符門檻的物件會從排行榜、物件列表與評比排行中隱藏。

## 4. 客觀資料欄位

### 4.1 基本識別

| key | 欄位 | 型別 | 說明 |
| --- | --- | --- | --- |
| `buildingName` | 建案/大樓名稱 | text | 方便辨識物件 |

| `district` | 行政區 | text | 查詢實價登錄與區域分類 |
| `businessZone` | 商圈/地標 | text | 例如高鐵站、藝文特區、百貨商圈 |

| key | 欄位 | 型別 | 說明 |

## 5. 頁面與功能

### 5.1 頂部控制列

功能：

1. 顯示系統名稱。
2. 切換策略預設：平衡模式、品牌優先、投報優先。
3. 匯入資料 CSV。
4. 匯出資料 CSV。
5. 切換頁面：物件管理、評比排行。

### 5.2 物件管理頁

左側面板：

1. 全域距離目標。
2. 全域期待指標。
3. 權重配置。
4. 即時排行榜。
5. CSV 欄位規格。

右側內容：

1. 候選物件卡片。
2. 原始客觀數據輸入。
3. 查實價登錄按鈕。
4. 共用測距。
5. 地圖預覽。
6. 財務效益自動計算。
7. 專家出價分析。
8. 策略維度綜合評分。

### 5.3 查實價登錄

第一階段採純前端導流與人工補登：

1. 使用物件的 `district` 與 `address` 組成查詢關鍵字。
2. 點擊「查實價登錄」後，系統嘗試將關鍵字複製到剪貼簿。
3. 開啟內政部實價登錄網站。
4. 使用者查詢後，手動填入：
   - `lastHistoricalPrice`
   - `historicalDate`
   - `marketSampleCount1Y`
   - `averageUnitPrice1Y`
   - `medianUnitPrice1Y`
   - `maxUnitPriceHistorical`
   - `minUnitPriceHistorical`
   - `averageUnitPriceHistorical`
   - `latestTransactionDate`

此設計不需要後端、不需要 API key，也避免爬蟲與 CORS 問題。

目前不建議讓前端直接爬取實價登錄網站，原因包括：

1. 查詢網站可能有 session、動態載入、查詢限制與版面變動。
2. 瀏覽器端會遇到 CORS 與反爬限制。
3. 地址、建案名稱與實價資料並非一對一，仍需資料清理。
4. 店面、住宅、辦公、含車位與特殊交易不可直接混算。

若未來要自動化，建議改採後端匯入官方 Open Data 或建立內部行情資料庫，再提供 API 給前端查詢。

### 5.4 評比排行頁

評比排行頁採轉置比較表：

1. 左側第一欄為「項目」。
2. 上方欄位為排序後物件：第 1 名、第 2 名、第 3 名。
3. 列順序：
   - 總分。
   - 總價。
   - 近一年中位單價、平均單價、歷史平均單價。
   - 周邊實登均價、樣本數、最近成交年月。
   - 市場估值價、建議下錨價、最高追價、開價偏離市場。
   - 不符條件。
   - 策略維度綜合評分。
   - 全域距離目標距離。
   - 財務計算值。
   - 所有 raw 客觀資料。
4. 項目列可拖拉對調順序。
5. 匯出排行對比表 CSV 時，會沿用使用者目前調整後的項目順序。

視覺規則：

| 狀態 | 呈現方式 |
| --- | --- |
| 數值最高 | 淡綠背景、深綠粗體 |
| 數值最低 | 淡紅背景、深紅粗體 |
| raw 資料有差異 | 整列淡黃背景、值加粗 |
| 不符條件物件 | 標示紅色提示文字或標籤 |

評比排行頁另提供「匯出排行對比表」CSV，格式與畫面一致。

## 6. 核心演算法

### 6.1 權重正規化

系統允許權重總和不等於 100。計分時必須先正規化。

```text
單項正規化權重 = 單項權重 / 權重總和 * 100
單項最終得分 = 物件該項評分 / 10 * 單項正規化權重
加權總分 = 所有單項最終得分加總
```

若權重總和為 0，實作上應使用 safeTotal 避免除以 0。

### 6.2 財務坪效

車位價格前處理：

```text
若 車位數量 > 0 且 車位價格為空或 0
則 計算用車位價格 = 200 萬
```

權狀單坪售價：

```text
(總價 - 計算用車位價格) / 建坪
```

實際登記單價：

```text
(總價 - 計算用車位價格) /
(1F樓板坪數 + 附屬坪數 + 夾層坪數 + 共有坪數)
```

核心坪效單價：

```text
若 挑高可夾層 == 是
分母 = 1F樓板坪數 * 2

若 挑高可夾層 == 否
分母 = 1F樓板坪數

核心坪效單價 = (總價 - 計算用車位價格) / 分母
```

### 6.3 全域門檻檢核

```text
若 targetMaxPrice > 0 且 totalPrice > targetMaxPrice
標示：預算超標

若 targetMinArea > 0 且 buildArea < targetMinArea
標示：坪數不足

若 targetMinWidth > 0 且 frontageWidth < targetMinWidth
標示：面寬不足
```

若使用者啟用「隱藏不符條件物件」，上述任一條件不符的物件不會出現在目前列表與排行中。

### 6.4 專家出價分析

此模組是議價參考，不是正式鑑價。

市場基礎單價採資料可信度排序。系統不直接爬蟲，而是使用使用者手動查詢後填入的第一階段行情統計資料。

```text
若 近一年中位單價 > 0 且 近一年樣本數 >= 5
市場基礎單價 = 近一年中位單價

否則若 近一年平均單價 > 0 且 近一年樣本數 >= 3
市場基礎單價 = 近一年平均單價

否則若 歷史平均單價 > 0
市場基礎單價 = 歷史平均單價

否則若 周邊實登均價 > 0
市場基礎單價 = 周邊實登均價

否則
市場基礎單價 = 權狀單坪售價 * 0.85
```

市場估值價不是用核心坪數直接推總價，而是以市場單價乘權狀建坪，再做店面條件修正。

```text
市場估值價 =
市場基礎單價 * 建坪 * 店面條件修正係數
+ 車位調整
- 硬體扣除
```

店面條件修正係數：

```text
店面條件修正係數 =
1
+ 角間溢價
+ 面寬溢價
+ 挑高溢價
+ 主幹道溢價
+ 採光溢價
- 屋齡折減
- 格局缺陷折減
```

目前系統採用的修正規則：

| 條件 | 修正 |
| --- | --- |
| 角間 | +5% |
| 面寬 >= 8 米 | +5% |
| 面寬 >= 6 米且 < 8 米 | +3% |
| 挑高可夾層 | +3% |
| 主幹道 | +3% |
| 1F 採光佳 | +2% |
| 屋齡 > 30 年 | -8% |
| 屋齡 > 20 年且 <= 30 年 | -3% |
| 格局不方正 | -4% |

修正係數最低限制為 0.75，避免過度折減。

硬體扣除：

```text
若 hd1 < 5
硬體扣除 = 建坪 * renovationDeductionPerPing

否則
硬體扣除 = 0
```

車位調整：

```text
若車位數量 > 0
車位調整 = 計算用車位價格

否則
車位調整 = 0
```

建議下錨價：

```text
建議下錨價 = 市場估值價 * 0.90
```

最高追價：

```text
若物件具備明顯溢價條件
最高追價 = 市場估值價 * 1.05

否則
最高追價 = 市場估值價 * 1.03
```

開價偏離市場：

```text
開價偏離市場 = (總價 - 市場估值價) / 市場估值價 * 100%
```

介面會呈現：

1. 該物件開價。
2. 市場估值價。
3. 市場基礎單價與採用來源。
4. 建議下錨價。
5. 最高追價。
6. 店面條件修正係數、車位調整、硬體扣除。
7. 文字評語。

## 7. CSV 匯入匯出規格

### 7.1 設計原則

CSV 必須滿足：

1. 使用者只按一次匯出完整資料。
2. 使用者只按一次匯入完整資料。
3. 全域設定與物件資料分離。
4. 缺漏欄位不造成系統錯誤。
5. Excel 開啟繁體中文不亂碼。

因此採用「單檔、多區段 CSV」。

所有匯出 CSV 皆需加入 UTF-8 BOM：

```js
const csv = "\uFEFF" + content;
```

### 7.2 完整資料 CSV 區段

```csv
#SECTION,GLOBAL_DISTANCE_TARGETS
目標序號,目標名稱,地址/地標
1,高鐵站,高鐵桃園站
2,捷運站,桃園捷運A19站

#SECTION,GLOBAL_CONSTRAINTS
參數,值,單位
targetMaxPrice,3500,萬
targetMinArea,35,坪
targetMinWidth,6,米
renovationDeductionPerPing,5,萬/坪

#SECTION,PROPERTIES
id,物件名稱,地址,總價(萬),建坪,...,距離-高鐵站(公尺),評分-商圈定位,...,市場估值價,建議下錨價,最高追價
1690001234567,物件A,桃園市中壢區...,3200,45,...,650,8,...,2950,2655,3038
```

### 7.3 `GLOBAL_DISTANCE_TARGETS`

| 欄位 | 說明 |
| --- | --- |
| 目標序號 | 1 到 5 |
| 目標名稱 | 高鐵站、住家、國小等 |
| 地址/地標 | Google Maps 可搜尋的地址或地標 |

### 7.4 `GLOBAL_CONSTRAINTS`

| 參數 | 說明 |
| --- | --- |
| `targetMaxPrice` | 預算上限，單位萬 |
| `targetMinArea` | 最低建坪 |
| `targetMinWidth` | 最低面寬，單位米 |
| `renovationDeductionPerPing` | 硬體評分低於 5 分時的每坪扣除額 |

### 7.5 `PROPERTIES`

欄位分類：

1. 基本欄位：`id`、`物件名稱`。
2. rawData 客觀資料欄位。
3. distances 距離欄位，例如 `距離-高鐵站(公尺)`。
4. scores 評分欄位，例如 `評分-商圈定位`。
5. derived 匯出時計算欄位，例如 `加權總分`、`核心坪效單價`、`市場估值價`、`建議下錨價`、`最高追價`。

匯入時 derived 欄位應忽略並重新計算。

### 7.6 評比排行對比表 CSV

此匯出是會議報表，不用於還原系統狀態。

格式：

```csv
項目,第 1 名 物件A,第 2 名 物件B
總分｜排序,78.50,72.20
總價(萬)｜價格與行情,3200.00,2800.00
近一年中位單價｜萬/坪,61.00,0.00
市場估值價｜專家出價分析,2950.00 萬,2600.00 萬
建議下錨價｜專家出價分析,2655.00 萬,2340.00 萬
不符條件｜門檻檢核,符合,預算超標
商圈定位｜權重 15%,8,9
距離-高鐵站｜高鐵桃園站,650.00 公尺,7800.00 公尺
```

CSV 不支援背景色，因此最高/最低顏色只存在於前端畫面。匯出的項目順序會跟隨使用者在評比排行頁拖拉後的目前順序。若未來要求 Excel 保留顏色，需改用 xlsx 函式庫。

### 7.7 匯入容錯

| 情況 | 處理 |
| --- | --- |
| 缺少 `id` | 自動產生 timestamp |
| 缺少物件名稱 | 使用「匯入物件 N」 |
| 數字欄位空白 | 使用 0 |
| 數字欄位無法解析 | 使用 0 |
| 評分缺漏 | 使用預設值 5 |
| 評分超出 0-10 | clamp 到 0-10 |
| 是/否欄位缺漏 | 使用「否」 |
| select 欄位值不合法 | 使用第一個選項 |
| 全域距離目標缺漏 | 保留現有設定或使用預設 |
| 全域門檻缺漏 | 保留現有設定 |
| derived 欄位存在 | 忽略並重新計算 |

## 8. UI/UX 設計規範

### 8.1 整體原則

這是決策工具，不是行銷頁。介面應：

1. 資訊密度高但不混亂。
2. 用面板、表格、分組欄位協助掃描。
3. 避免過度裝飾。
4. 讓使用者快速比較、快速發現差異。

### 8.2 權重與門檻

權重使用 slider，旁邊顯示百分比。權重總和不等於 100 時只警示，不阻止操作。

全域門檻使用數字輸入。門檻不應直接混入加權總分，因為門檻是硬性條件，策略評分是偏好排序，兩者需分離。

### 8.3 刪除防呆

禁止使用 `window.confirm()`。應使用自訂 Modal：

1. 背景半透明遮罩。
2. 顯示物件名稱。
3. 明確的刪除與取消按鈕。

### 8.4 評比排行

採用轉置表格：

1. 項目放在列。
2. 物件放在欄。
3. 物件欄依總分排序。
4. 第一欄固定，便於橫向掃描。

視覺提示以背景色為主、粗體為輔，因為表格掃描時背景色比單純文字顏色更有效。

### 8.5 行動端

行動端排列：

1. 全域距離目標。
2. 全域期待指標。
3. 權重控制。
4. 排行榜。
5. 物件卡片。

range input 觸控高度需至少 44px。

## 11. 房仲與展店專家的使用建議

### 11.1 不要只看總分

總分是排序工具，不是唯一答案。建議同時查看：

1. 不符條件標籤。
2. 核心坪效單價。
3. 市場估值價、建議下錨價與最高追價。
4. 停車與距離條件。
5. 面寬、樓高、採光。
6. 是否有無法靠裝修改善的缺陷。

### 11.2 門檻用於一票否決，權重用於偏好排序

例如：

1. 預算超標：可能直接排除，或作為議價前提。
2. 坪數不足：可能不符合展示或倉儲需求。
3. 面寬不足：可能影響招牌、展示與品牌形象。

因此門檻不應被加權分數稀釋，而應獨立標示。

### 11.3 實價資料要記錄年月

不動產行情會隨時間變化。只填均價不夠，應同時記錄行情年月、近一年樣本數、平均單價與中位單價。若樣本數不足，中位數與平均數的參考價值都會下降。

### 11.4 建議買價是議價參考

市場估值價、建議下錨價與最高追價不等於正式鑑價。它主要用於：

1. 決策會議建立共同討論基準。
2. 協助房仲設定斡旋起點。
3. 快速辨識開價是否偏離市場行情。

### 12.1 資料持久化

長期：

### 12.2 自動化實價登錄

短期已採導流方案。若未來導入後端，可評估：

1. 地址解析。
2. 半徑查詢。
3. 屋齡與建物型態過濾。
4. 最近一年交易資料。
5. 自動計算近一年中位數、平均、最高、最低、樣本數與最近成交年月。
6. 將行情信心分級為 none、low、medium、high。

不建議純前端爬取第三方平台，原因包括 CORS、服務條款、穩定性與資料清理成本。

### 12.3 xlsx 報表

## 13. 目前版本限制

1. 無後端，刷新後若未加 localStorage 會遺失狀態。
2. 地圖距離需人工回填。
3. 實價登錄採導流與人工補登，尚未自動串接。
4. CSV 可保存資料，但不是多人協作格式。
5. 建議買價是估算模型，不是正式鑑價。
6. 財務公式尚未包含稅費、貸款利率、租金收益與機會成本。
7. 未處理商圈人口、車流、人流等外部資料。

## 14. 結論

本系統適合用作店面初篩與決策會議工具。新版已從單純評分表提升為具備硬性門檻、行情時間維度、專家議價分析與會議報表匯出的選址決策系統。

對開發團隊而言，最重要的是維持資料邊界：

1. 全域距離放 `GLOBAL_DISTANCE_TARGETS`。
2. 全域門檻放 `GLOBAL_CONSTRAINTS`。
3. 物件資料放 `PROPERTIES`。
4. 計算值匯出可讀，但匯入時應重新計算。
5. 評比排行 CSV 是報表，不是系統還原資料。

對房仲與展店專家而言，最重要的是固定評分標準、填入行情年月，並善用評比排行頁找出真正影響決策的差異項目。

# 智慧居家總店選址評估系統說明手冊

版本：v1.1  
更新日期：2026-06-23  
適用對象：前端工程師、全端工程師、產品經理、UI/UX 設計師、房仲專家、展店顧問  
系統型態：純前端 Web App，無後端、無資料庫、無付費地圖 API 即可運作

## 1. 系統目的

本系統用於協助智慧居家、展示中心、總店或旗艦店進行候選店面比較。它把選址決策中常見的主觀評分、客觀條件、財務坪效、交通距離、硬性門檻與專家出價分析整合為一套可重複使用的量化流程。

系統核心目標：

1. 同時比較多個候選店面。
2. 將房仲或展店顧問提供的物件資訊結構化。
3. 讓決策者調整策略權重，例如平衡模式、品牌優先、投報優先。
4. 讓使用者統一設定全域距離目標，例如住家、國小、百貨公司、高鐵站、捷運站。
5. 讓使用者設定全域硬性門檻，例如預算上限、最低建坪、最低面寬。
6. 即時計算加權總分、財務坪效、市場估值價、建議下錨價、最高追價。
7. 在評比排行頁以轉置表格呈現所有物件差異。
8. 評比排行項目列可由使用者拖拉排序，也可依買方需求顯示或隱藏。
9. 以單一 CSV 保存完整評估案，並可再次匯入繼續評估。
10. 額外支援評比排行對比表 CSV 匯出，供決策會議使用。

## 2. 技術架構

| 類別 | 建議 |
| --- | --- |
| 前端框架 | React + Vite |
| 狀態管理 | React Hooks：`useState`、`useMemo`、`useRef` |
| 樣式 | CSS 或 Tailwind CSS，需支援 RWD |
| 圖示 | Lucide React 或等效 SVG icon |
| 地圖 | Google Maps Embed iframe 與 Google Maps Intent URL |
| 實價登錄 | 第一階段採導流與人工補登行情統計，不做前端爬蟲 |
| CSV | 原生 JavaScript Blob API + File API |
| 帳號與雲端資料 | Firebase Authentication + Firestore |
| 後端 | 不自建後端；使用 Firebase 代管登入與資料隔離 |

Firebase 未設定時，系統仍可使用全部本機評估、CSV 匯入與 CSV 匯出功能；只是不提供帳號登入與雲端資料集保存。

## 3. 核心資料模型

系統主要維護六類資料：

1. `weights`：策略評分權重。
2. `properties`：候選物件資料陣列。
3. `distanceTargets`：全域距離目標。
4. `constraints`：全域期待指標與硬性門檻。
5. `cloud datasets`：登入使用者儲存在 Firestore 的多份評估資料。
6. UI state：頁面切換、Accordion、刪除 Modal、是否隱藏不符條件物件等。

### 3.1 權重資料 `weights`

權重使用 key-value 物件保存。每個值建議為 0 到 50 的整數。

```js
{
  loc1: 15,
  loc2: 10,
  fac1: 10,
  fac2: 10,
  spc1: 10,
  spc2: 15,
  fin1: 10,
  fin2: 10,
  hd1: 10
}
```

| key | 名稱 | 群組 | 用途 |
| --- | --- | --- | --- |
| `loc1` | 商圈定位 | 區位 | 商圈、地段、品牌曝光與客群契合度 |
| `loc2` | 接駁停車 | 區位 | 停車、捷運、高鐵、交通便利性 |
| `fac1` | 能見度 | 店況 | 門面、招牌、車流與人流可見性 |
| `fac2` | 格局方正 | 店況 | 展示、動線、施工便利性 |
| `spc1` | 挑高條件 | 空間 | 夾層、展示牆、體驗空間 |
| `spc2` | 坪數採光 | 空間 | 使用坪效、採光與開放感 |
| `fin1` | 總價預算 | 財務 | 是否符合購置預算 |
| `fin2` | 單價急售 | 財務 | 單價、議價空間、急售機會 |
| `hd1` | 既有硬體 | 硬體 | 空調、水電、消防、裝修可沿用程度 |

### 3.2 全域距離目標 `distanceTargets`

距離目標由使用者設定一次，所有物件共用。

```js
[
  { id: "target1", name: "高鐵站", address: "高鐵桃園站" },
  { id: "target2", name: "捷運站", address: "桃園捷運A19站" },
  { id: "target3", name: "停車場", address: "附近停車場" },
  { id: "target4", name: "自訂基準點", address: "" }
]
```

設計理由：

1. 住家、學校、百貨公司、指定客戶公司等目標會因決策目的而變。
2. 這些資料屬於評估案全域條件，不應混入單一物件資料。
3. CSV 匯出時獨立成 `GLOBAL_DISTANCE_TARGETS` 區段。

目前建議最多 5 個距離目標。

### 3.3 全域期待指標 `constraints`

`constraints` 是一次設定、套用到所有物件的硬性門檻與估價參數。

```js
{
  targetMaxPrice: 0,
  targetMinArea: 0,
  targetMinWidth: 0,
  renovationDeductionPerPing: 5
}
```

| key | 名稱 | 單位 | 說明 |
| --- | --- | --- | --- |
| `targetMaxPrice` | 預算上限 | 萬 | 若物件總價超過此值，標示 `預算超標` |
| `targetMinArea` | 最低建坪 | 坪 | 若建坪低於此值，標示 `坪數不足` |
| `targetMinWidth` | 最低面寬 | 米 | 若面寬低於此值，標示 `面寬不足` |
| `renovationDeductionPerPing` | 硬體扣除額 | 萬/坪 | 當硬體評分低於 5 分時，用於建議買價扣除 |

當門檻值為 0 時，代表不啟用該門檻。

系統提供「隱藏不符條件物件」切換：

1. 關閉時：所有物件仍顯示，但不符者顯示紅色標籤。
2. 開啟時：不符門檻的物件會從排行榜、物件列表與評比排行中隱藏。

## 4. 客觀資料欄位

### 4.1 基本識別

| key | 欄位 | 型別 | 說明 |
| --- | --- | --- | --- |
| `buildingName` | 建案/大樓名稱 | text | 方便辨識物件 |
| `address` | 地址 | text | 地圖預覽、測距與實價查詢關鍵字 |
| `district` | 行政區 | text | 查詢實價登錄與區域分類 |
| `businessZone` | 商圈/地標 | text | 例如高鐵站、藝文特區、百貨商圈 |
| `coverPhotoUrl` | 封面照網址 | text | 物件主要照片網址；評比排行欄位標題會顯示此圖或照片清單第一張 |
| `photoUrls` | 照片網址清單 | textarea | 多張照片網址，使用分號 `;` 或換行分隔；CSV 只保存網址，不保存圖片檔 |

### 4.2 地理位置與環境

| key | 欄位 | 型別 | 說明 |
| --- | --- | --- | --- |
| `isMainRoad` | 主幹道 | 是/否 | 是否位於主要道路 |
| `direction` | 座向 | select | 東、西、南、北、東南等 |
| `nearbyParkingNote` | 附近停車描述 | text | 停車場、路邊停車、裝卸便利性 |
| `environmentNote` | 環境備註 | text | 鄰近業種、街廓品質、人流特性 |

### 4.3 財務與坪效

| key | 欄位 | 型別 | 說明 |
| --- | --- | --- | --- |
| `totalPrice` | 總價(萬) | number | 物件開價或總價 |
| `buildArea` | 建坪 | number | 權狀建坪 |
| `firstFloorArea` | 1F樓板坪數 | number | 店面核心展示面積 |
| `attachedArea` | 附屬坪數 | number | 陽台、附屬建物等 |
| `hasMezzanine` | 挑高可夾層 | 是/否 | 影響核心坪效與有效坪數 |
| `mezzanineArea` | 夾層坪數 | number | 可使用或登記夾層面積 |
| `sharedArea` | 共有坪數 | number | 公設或共有 |
| `landArea` | 土地坪數 | number | 土地持分或土地面積 |
| `parkingCount` | 車位數量 | number | 車位數 |
| `parkingArea` | 車位坪數 | number | 車位面積 |
| `parkingPrice` | 車位價格(萬) | number | 若車位數大於 0 且為空/0，計算時預設 200 |
| `age` | 屋齡 | number | 建物年限 |
| `actualTransactionTotalPrice` | 實登總價(萬) | number | 實價登錄揭露的成交總價，通常包含建物與車位 |
| `actualTransactionParkingPrice` | 實登車位價格(萬) | number | 實價登錄揭露或人工拆分的車位價格；空白時不自動套用賣方車位價格 |
| `lastHistoricalPrice` | 周邊實登均價(萬/坪) | number | 使用者查詢後手動補登 |
| `historicalDate` | 實登行情年月 | text | 行情參考年月，例如 2026-05 |
| `marketSampleCount1Y` | 近一年樣本數 | number | 第一階段由使用者查詢後手動補登 |
| `averageUnitPrice1Y` | 近一年平均單價(萬/坪) | number | 近一年成交平均 |
| `medianUnitPrice1Y` | 近一年中位單價(萬/坪) | number | 近一年成交中位數，估價優先度最高 |
| `maxUnitPriceHistorical` | 歷史最高單價(萬/坪) | number | 歷史行情區間上緣 |
| `minUnitPriceHistorical` | 歷史最低單價(萬/坪) | number | 歷史行情區間下緣 |
| `averageUnitPriceHistorical` | 歷史平均單價(萬/坪) | number | 歷史長期均價 |
| `latestTransactionDate` | 最近成交年月 | text | 最近一筆參考成交時間 |

### 4.4 店面條件

| key | 欄位 | 型別 | 說明 |
| --- | --- | --- | --- |
| `isCornerLot` | 角間 | 是/否 | 是否具雙面曝光 |
| `frontageWidth` | 面寬(米) | number | 招牌、櫥窗、展示面 |
| `depth` | 深度(米) | number | 展示動線與儲藏配置 |
| `roadWidth` | 道路寬(米) | number | 車流、可見性、裝卸便利 |
| `isSquare` | 格局方正 | 是/否 | 是否利於展示與裝修 |
| `ceilingHeight` | 樓高(米) | number | 夾層與展示牆可行性 |
| `firstFloorLight` | 1F採光 | select | 佳、普通、弱 |
| `secondFloorLight` | 2F採光 | select | 佳、普通、弱、無2F |
| `hardwareNote` | 既有硬體備註 | text | 空調、水電、消防、裝修狀態 |
| `memo` | 備註 | textarea | 其他判斷 |

## 5. 頁面與功能

### 5.1 頂部控制列

功能：

1. 顯示系統名稱。
2. Firebase 登入：Google 登入、Email/Password 登入、註冊、登出。
3. 切換策略預設：平衡模式、品牌優先、投報優先。
4. 匯入資料 CSV。
5. 匯出資料 CSV。
6. 切換頁面：物件管理、評比排行。

### 5.2 物件管理頁

左側面板：

1. 雲端資料集管理。
2. 全域距離目標。
3. 全域期待指標。
4. 權重配置。
5. 即時排行榜。
6. CSV 欄位規格。

右側內容：

1. 候選物件卡片。
2. 原始客觀數據輸入。
3. 查實價登錄按鈕。
4. 共用測距。
5. 地圖預覽。
6. 物件照片預覽：封面大圖、縮圖列、點圖放大。
7. 財務效益自動計算。
8. 專家出價分析。
9. 策略維度綜合評分。

照片顯示規則：

1. `coverPhotoUrl` 優先作為封面照。
2. 若未填封面照，使用 `photoUrls` 第一張作為封面。
3. 物件卡片顯示封面大圖與縮圖列，點縮圖切換卡片大圖。
4. 點卡片大圖會開啟照片放大 Modal。
5. 放大 Modal 支援上一張、下一張與縮圖切換。
6. 評比排行頁的物件欄位標題顯示封面縮圖，點擊同樣開啟放大 Modal。
7. 圖片網址必須是瀏覽器可公開讀取的 URL；若雲端圖片未開權限，系統無法代替使用者取得圖片。

### 5.3 帳號與雲端資料集

目前版本採 Firebase：

1. Firebase Authentication：
   - Google 登入。
   - Email/Password 註冊與登入。
2. Firestore：
   - 每位使用者有自己的資料集清單。
   - 使用者可儲存目前評估資料。
   - 使用者可另存新檔，保留多份不同檔名資料。
   - 使用者可讀取或刪除自己的資料。
3. CSV 匯入後：
   - 未登入：只匯入到目前瀏覽器畫面。
   - 已登入：匯入後自動用 CSV 檔名另存到該使用者雲端資料集。
4. 下次登入：
   - 系統自動讀取該使用者最近更新的一份資料。

Firestore 路徑：

```text
users/{uid}/datasets/{datasetId}
```

資料集內容：

```js
{
  name: "青埔總店候選案",
  payload: {
    properties,
    distanceTargets,
    constraints,
    weights,
    hideNonCompliant,
    comparisonRowOrder,
    hiddenComparisonRows,
    pinnedComparisonRows,
    comparisonSort
  },
  createdAt,
  updatedAt
}
```

使用者資料隔離必須由 Firestore Security Rules 控制。建議規則：

```js
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /users/{userId}/datasets/{datasetId} {
      allow read, write: if request.auth != null && request.auth.uid == userId;
    }
  }
}
```

Firebase Web App 需在 `.env` 設定：

```text
VITE_FIREBASE_API_KEY=...
VITE_FIREBASE_AUTH_DOMAIN=...
VITE_FIREBASE_PROJECT_ID=...
VITE_FIREBASE_STORAGE_BUCKET=...
VITE_FIREBASE_MESSAGING_SENDER_ID=...
VITE_FIREBASE_APP_ID=...
```

### 5.4 查實價登錄

第一階段採純前端導流與人工補登：

1. 使用物件的 `district` 與 `address` 組成查詢關鍵字。
2. 點擊「查實價登錄」後，系統嘗試將關鍵字複製到剪貼簿。
3. 開啟內政部實價登錄網站。
4. 使用者查詢後，手動填入：
   - `lastHistoricalPrice`
   - `historicalDate`
   - `marketSampleCount1Y`
   - `averageUnitPrice1Y`
   - `medianUnitPrice1Y`
   - `maxUnitPriceHistorical`
   - `minUnitPriceHistorical`
   - `averageUnitPriceHistorical`
   - `latestTransactionDate`

此設計不需要後端、不需要 API key，也避免爬蟲與 CORS 問題。

目前不建議讓前端直接爬取實價登錄網站，原因包括：

1. 查詢網站可能有 session、動態載入、查詢限制與版面變動。
2. 瀏覽器端會遇到 CORS 與反爬限制。
3. 地址、建案名稱與實價資料並非一對一，仍需資料清理。
4. 店面、住宅、辦公、含車位與特殊交易不可直接混算。

若未來要自動化，建議改採後端匯入官方 Open Data 或建立內部行情資料庫，再提供 API 給前端查詢。

### 5.4 評比排行頁

評比排行頁採轉置比較表：

1. 左側第一欄為「項目」。
2. 上方欄位為排序後物件：第 1 名、第 2 名、第 3 名。
3. 列順序：
   - 總分。
   - 總價 / 實登或估值。
   - 近一年中位單價、平均單價、歷史平均單價。
   - 周邊實登均價、樣本數、最近成交年月。
   - 總價全坪單價 / 實登、建物扣車位單價 / 實登基準、可擴充坪效 / 實登基準、核心坪效 / 實登基準。
   - 市場估值價、建議下錨價、最高追價、開價偏離市場。
   - 不符條件。
   - 策略維度綜合評分。
   - 全域距離目標距離。
   - 所有 raw 客觀資料。
4. 項目列可拖拉對調順序。
5. 每個項目可顯示或隱藏，避免不同買方不關心的欄位佔用會議畫面。
6. 每個項目可錨定。錨定後會出現在頁面級錨定摘要區，並在使用者往下捲動網頁時持續固定於視窗上方，適合固定總價、建坪、坪效、距離等高頻比較列。
7. 數值型項目可作為物件欄位排序依據，例如總價、建坪、面寬、距離、評分、估值與數字型 raw 欄位。
8. 項目名稱提供滑鼠 hover tooltip。使用者將滑鼠移到項目名稱時，需顯示該項目的資料來源、公式或判讀方式。
9. 匯出排行對比表 CSV 時，會沿用使用者目前調整後的項目順序與物件排序，並排除已隱藏項目。

價格資訊顯示規則：

| 項目 | 第一行 | 第二行 | 第三行 |
| --- | --- | --- | --- |
| 總價 / 實登或估值 | 賣家開價，單位萬 | 若有實登總價則顯示實登總價，否則顯示市場估值價 | 開價相對實登總價或市場估值的差異百分比 |
| 總價全坪單價 / 實登 | 賣方總價全坪單價，單位萬/坪 | 實登總價全坪單價，單位萬/坪 | 賣方全坪單價相對實登全坪單價的差異百分比 |
| 建物扣車位單價 / 實登基準 | 賣方建物扣車位單價，單位萬/坪 | 若有實登總價則顯示實登建物扣車位單價，否則顯示市場基準單價 | 賣方單價相對實登或市場基準的差異百分比 |
| 可擴充坪效 / 實登基準 | 賣方可擴充使用坪效單價，單位萬/坪 | 若有實登總價則顯示實登可擴充使用坪效單價，否則顯示市場基準單價 | 賣方單價相對實登或市場基準的差異百分比 |
| 核心坪效 / 實登基準 | 賣方核心坪效單價，單位萬/坪 | 若有實登總價則顯示實登核心坪效單價，否則顯示市場基準單價 | 賣方核心坪效單價相對實登或市場基準的差異百分比 |

此處不採用 `賣家開價(實價登錄)` 的括弧寫法，原因是括弧在多物件橫向比較時不易掃描，也不利於同時呈現差異百分比。上下雙層加差異標籤更適合決策會議閱讀。

實登總價拆分規則：

```text
實登建物價格 = 實登總價 - 實登車位價格
實登總價全坪單價 = 實登總價 / 建坪
實登建物扣車位單價 = 實登建物價格 / (建坪 - 車位坪數)
實登可擴充使用坪效單價 = 實登建物價格 / 可擴充使用面積
實登核心坪效單價 = 實登建物價格 / 核心坪效面積
```

若 `實登車位價格(萬)` 缺漏，系統不會自動套用賣方 `車位價格(萬)`，因為兩者可能來自不同時間點或不同交易條件。

項目註解規則：

| 項目類型 | 註解內容 |

| 實登拆分欄位 | 說明 `實登建物價格 = 實登總價 - 實登車位價格` 與相關單價分母 |
| 價格合併欄位 | 說明第一行、第二行、第三行各自代表的資料 |
| 專家出價欄位 | 說明市場估值價、建議下錨價、最高追價的公式 |
| 策略評分欄位 | 說明 0-10 分評分會依目前權重正規化後計入總分 |
| 距離欄位 | 說明距離目標名稱、目標地址與距離單位 |

視覺規則：

| 狀態 | 呈現方式 |
| --- | --- |
| 數值最高 | 淡綠背景、深綠粗體 |
| 數值最低 | 淡紅背景、深紅粗體 |

## 6. 核心演算法

### 6.1 權重正規化

系統允許權重總和不等於 100。計分時必須先正規化。

```text
單項正規化權重 = 單項權重 / 權重總和 * 100
單項最終得分 = 物件該項評分 / 10 * 單項正規化權重
加權總分 = 所有單項最終得分加總
```

若權重總和為 0，實作上應使用 safeTotal 避免除以 0。

### 6.2 財務坪效

車位價格前處理：

```text
若 車位數量 > 0 且 車位價格為空或 0
則 計算用車位價格 = 200 萬
```

總價全坪單價：

```text
總價 / 建坪
```

建物扣車位單價：

```text
(總價 - 計算用車位價格) / (建坪 - 車位坪數)
```

可擴充使用坪效單價：

```text
若 挑高可夾層 == 是
分母 = 1F樓板坪數 * 2 + 附屬坪數 + 共有坪數

若 挑高可夾層 == 否
分母 = 1F樓板坪數 + 附屬坪數 + 共有坪數

可擴充使用坪效單價 = (總價 - 計算用車位價格) / 分母
```

核心坪效單價：

```text
若 挑高可夾層 == 是
分母 = 1F樓板坪數 * 2

若 挑高可夾層 == 否
分母 = 1F樓板坪數

核心坪效單價 = (總價 - 計算用車位價格) / 分母
```

`可擴充使用坪效單價` 不再把 `夾層坪數` 加入分母，因為當 1F 可做滿夾層時，系統直接以 `1F樓板坪數 * 2` 估算可用面積，避免既有夾層坪數被重複計算。

實登換算單價：

```text
實登建物價格 = 實登總價 - 實登車位價格
實登總價全坪單價 = 實登總價 / 建坪
實登建物扣車位單價 = 實登建物價格 / (建坪 - 車位坪數)
實登可擴充使用坪效單價 = 實登建物價格 / 可擴充使用坪效分母
實登核心坪效單價 = 實登建物價格 / 核心坪效分母
```

實登車位價格缺漏時不自動套用賣方車位價格，避免把開價條件誤混到成交資料中。

### 6.3 全域門檻檢核

```text
若 targetMaxPrice > 0 且 totalPrice > targetMaxPrice
標示：預算超標

若 targetMinArea > 0 且 buildArea < targetMinArea
標示：坪數不足

若 targetMinWidth > 0 且 frontageWidth < targetMinWidth
標示：面寬不足
```

若使用者啟用「隱藏不符條件物件」，上述任一條件不符的物件不會出現在目前列表與排行中。

### 6.4 專家出價分析

此模組是議價參考，不是正式鑑價。

市場基礎單價採資料可信度排序。系統不直接爬蟲，而是使用使用者手動查詢後填入的第一階段行情統計資料。

```text
若 近一年中位單價 > 0 且 近一年樣本數 >= 5
市場基礎單價 = 近一年中位單價

否則若 近一年平均單價 > 0 且 近一年樣本數 >= 3
市場基礎單價 = 近一年平均單價

否則若 實登建物扣車位單價 > 0
市場基礎單價 = 實登建物扣車位單價

否則若 歷史平均單價 > 0
市場基礎單價 = 歷史平均單價

否則若 周邊實登均價 > 0
市場基礎單價 = 周邊實登均價

否則
市場基礎單價 = 建物扣車位單價 * 0.85
```

市場估值價不是用核心坪數直接推總價，而是以市場單價乘「建坪扣車位坪數」後，再做店面條件修正並加回車位調整。

```text
市場估值價 =
市場基礎單價 * (建坪 - 車位坪數) * 店面條件修正係數
+ 車位調整
- 硬體扣除
```

店面條件修正係數：

```text
店面條件修正係數 =
1
+ 角間溢價
+ 面寬溢價
+ 挑高溢價
+ 主幹道溢價
+ 採光溢價
- 屋齡折減
- 格局缺陷折減
```

目前系統採用的修正規則：

| 條件 | 修正 |
| --- | --- |
| 角間 | +5% |
| 面寬 >= 8 米 | +5% |
| 面寬 >= 6 米且 < 8 米 | +3% |
| 挑高可夾層 | +3% |
| 主幹道 | +3% |
| 1F 採光佳 | +2% |
| 屋齡 > 30 年 | -8% |
| 屋齡 > 20 年且 <= 30 年 | -3% |
| 格局不方正 | -4% |

修正係數最低限制為 0.75，避免過度折減。

硬體扣除：

```text
若 hd1 < 5
硬體扣除 = 建坪 * renovationDeductionPerPing

否則
硬體扣除 = 0
```

車位調整：

```text
若車位數量 > 0
車位調整 = 計算用車位價格

否則
車位調整 = 0
```

建議下錨價：

```text
建議下錨價 = 市場估值價 * 0.90
```

最高追價：

```text
若物件具備明顯溢價條件
最高追價 = 市場估值價 * 1.05

否則
最高追價 = 市場估值價 * 1.03
```

開價偏離市場：

```text
開價偏離市場 = (總價 - 市場估值價) / 市場估值價 * 100%
```

介面會呈現：

1. 該物件開價。
2. 市場估值價。
3. 市場基礎單價與採用來源。
4. 建議下錨價。
5. 最高追價。
6. 店面條件修正係數、車位調整、硬體扣除。
7. 文字評語。

## 7. CSV 匯入匯出規格

### 7.1 設計原則

CSV 必須滿足：

1. 使用者只按一次匯出完整資料。
2. 使用者只按一次匯入完整資料。
3. 全域設定與物件資料分離。
4. 缺漏欄位不造成系統錯誤。
5. Excel 開啟繁體中文不亂碼。

因此採用「單檔、多區段 CSV」。

所有匯出 CSV 皆需加入 UTF-8 BOM：

```js
const csv = "\uFEFF" + content;
```

### 7.2 完整資料 CSV 區段

```csv
#SECTION,GLOBAL_DISTANCE_TARGETS
目標序號,目標名稱,地址/地標
1,高鐵站,高鐵桃園站
2,捷運站,桃園捷運A19站

#SECTION,GLOBAL_CONSTRAINTS
參數,值,單位
targetMaxPrice,3500,萬
targetMinArea,35,坪
targetMinWidth,6,米
renovationDeductionPerPing,5,萬/坪

#SECTION,PROPERTIES
id,物件名稱,地址,總價(萬),車位價格(萬),實登總價(萬),實登車位價格(萬),建坪,...,距離-高鐵站(公尺),評分-商圈定位,...,市場估值價,建議下錨價,最高追價
1690001234567,物件A,桃園市中壢區...,3200,200,3000,200,45,...,650,8,...,2950,2655,3038
```

### 7.3 `GLOBAL_DISTANCE_TARGETS`

| 欄位 | 說明 |
| --- | --- |
| 目標序號 | 1 到 5 |
| 目標名稱 | 高鐵站、住家、國小等 |
| 地址/地標 | Google Maps 可搜尋的地址或地標 |

### 7.4 `GLOBAL_CONSTRAINTS`

| 參數 | 說明 |
| --- | --- |
| `targetMaxPrice` | 預算上限，單位萬 |
| `targetMinArea` | 最低建坪 |
| `targetMinWidth` | 最低面寬，單位米 |
| `renovationDeductionPerPing` | 硬體評分低於 5 分時的每坪扣除額 |

### 7.5 `PROPERTIES`

欄位分類：

1. 基本欄位：`id`、`物件名稱`。
2. rawData 客觀資料欄位，包含 `coverPhotoUrl` 與 `photoUrls`。
3. distances 距離欄位，例如 `距離-高鐵站(公尺)`。
4. scores 評分欄位，例如 `評分-商圈定位`。
5. derived 匯出時計算欄位，例如 `加權總分`、`實登建物價格`、`總價全坪單價`、`建物扣車位單價`、`可擴充使用坪效單價`、`核心坪效單價`、`市場估值價`、`建議下錨價`、`最高追價`。

匯入時 derived 欄位應忽略並重新計算。

價格欄位匯入原則：

| 欄位 | 用途 |
| --- | --- |
| `總價(萬)` | 賣方開價或目前欲比較的交易總價 |
| `車位價格(萬)` | 賣方開價中拆出的車位價格；若車位數大於 0 且空白，計算時預設 200 |
| `實登總價(萬)` | 實價登錄成交總價，包含建物與車位 |
| `實登車位價格(萬)` | 實價登錄成交總價中拆出的車位價格；缺漏時不自動補 200 |

照片欄位匯入原則：

| 欄位 | 用途 |
| --- | --- |
| `封面照網址` | 物件封面照；空白時改用照片網址清單第一張 |
| `照片網址清單` | 多張照片網址，以分號 `;` 或換行分隔 |

CSV 不保存圖片檔本體，也不保存 Base64 圖片。這樣可避免 CSV 檔案過大、匯入失敗或 GitHub Pages 載入緩慢。

### 7.6 評比排行對比表 CSV

此匯出是會議報表，不用於還原系統狀態。

格式：

```csv
項目,第 1 名 物件A,第 2 名 物件B
總分｜排序,78.50,72.20
總價 / 實登或估值｜賣家開價 vs 實登總價/市場估值,3200.00 / 3000.00 / 6.67%,2800.00 / 2600.00 / 7.69%
總價全坪單價 / 實登｜萬/坪,71.11 / 66.67,73.68 / 0.00
近一年中位單價｜萬/坪,61.00,0.00
建物扣車位單價 / 實登基準｜萬/坪,65.12 / 60.47,73.68 / 58.00
可擴充坪效 / 實登基準｜萬/坪,71.79 / 66.67,84.85 / 58.00
核心坪效 / 實登基準｜萬/坪,57.14 / 56.00,62.22 / 58.00
市場估值價｜專家出價分析,2950.00 萬,2600.00 萬
建議下錨價｜專家出價分析,2655.00 萬,2340.00 萬
不符條件｜門檻檢核,符合,預算超標
商圈定位｜權重 15%,8,9
距離-高鐵站｜高鐵桃園站,650.00 公尺,7800.00 公尺
```

CSV 不支援背景色，因此最高/最低、價差最大/最小與錨定列顏色只存在於前端畫面。匯出的項目順序會跟隨使用者在評比排行頁拖拉後的目前順序，已隱藏項目不匯出，物件欄位排序則跟隨目前選用的排序項目。若未來要求 Excel 保留顏色，需改用 xlsx 函式庫。

### 7.7 匯入容錯

| 情況 | 處理 |
| --- | --- |
| 缺少 `id` | 自動產生 timestamp |
| 缺少物件名稱 | 使用「匯入物件 N」 |
| 數字欄位空白 | 使用 0 |
| 數字欄位無法解析 | 使用 0 |
| 評分缺漏 | 使用預設值 5 |
| 評分超出 0-10 | clamp 到 0-10 |
| 是/否欄位缺漏 | 使用「否」 |
| select 欄位值不合法 | 使用第一個選項 |
| 全域距離目標缺漏 | 保留現有設定或使用預設 |
| 全域門檻缺漏 | 保留現有設定 |
| derived 欄位存在 | 忽略並重新計算 |

## 8. UI/UX 設計規範

### 8.1 整體原則

這是決策工具，不是行銷頁。介面應：

1. 資訊密度高但不混亂。
2. 用面板、表格、分組欄位協助掃描。
3. 避免過度裝飾。
4. 讓使用者快速比較、快速發現差異。

### 8.2 權重與門檻

權重使用 slider，旁邊顯示百分比。權重總和不等於 100 時只警示，不阻止操作。

全域門檻使用數字輸入。門檻不應直接混入加權總分，因為門檻是硬性條件，策略評分是偏好排序，兩者需分離。

### 8.3 刪除防呆

禁止使用 `window.confirm()`。應使用自訂 Modal：

1. 背景半透明遮罩。
2. 顯示物件名稱。
3. 明確的刪除與取消按鈕。

### 8.4 評比排行

採用轉置表格：

1. 項目放在列。
2. 物件放在欄。
3. 物件欄依總分排序。
4. 物件欄標題顯示封面縮圖，點擊可開啟照片放大 Modal。
5. 第一欄固定，便於橫向掃描。
6. 價格類項目採上下雙層顯示，避免賣家開價、市場估值與單價基準拆成過多列。
7. 項目列提供勾選顯示/隱藏，讓不同買方可保留與自身決策最相關的資訊。
8. 項目列提供錨定功能，被錨定項目會另行顯示於頁面級 sticky 摘要區，避免網頁下捲後失去價格或坪數基準。
9. 項目列提供排序功能，只有具備 `getNumeric` 的數值型項目可排序；文字欄位不啟用排序按鈕。
10. 項目名稱提供 hover tooltip。所有由 CSV 原始資料推估而來的欄位、合併顯示欄位與自定義專家分析欄位都必須有 `description`。

視覺提示以背景色為主、粗體為輔，因為表格掃描時背景色比單純文字顏色更有效。

### 8.5 行動端

行動端排列：

1. 全域距離目標。
2. 全域期待指標。

## 11. 房仲與展店專家的使用建議

### 11.1 不要只看總分

總分是排序工具，不是唯一答案。建議同時查看：

1. 不符條件標籤。
2. 核心坪效單價。
3. 市場估值價、建議下錨價與最高追價。
4. 停車與距離條件。
5. 面寬、樓高、採光。
6. 是否有無法靠裝修改善的缺陷。

### 11.2 門檻用於一票否決，權重用於偏好排序

例如：

1. 預算超標：可能直接排除，或作為議價前提。
2. 坪數不足：可能不符合展示或倉儲需求。
3. 面寬不足：可能影響招牌、展示與品牌形象。

因此門檻不應被加權分數稀釋，而應獨立標示。

### 11.3 實價資料要記錄年月

不動產行情會隨時間變化。只填均價不夠，應同時記錄行情年月、近一年樣本數、平均單價與中位單價。若樣本數不足，中位數與平均數的參考價值都會下降。

### 11.4 建議買價是議價參考

市場估值價、建議下錨價與最高追價不等於正式鑑價。它主要用於：

1. 決策會議建立共同討論基準。
2. 協助房仲設定斡旋起點。
3. 快速辨識開價是否偏離市場行情。

## 15. 最新版本補充：全域房貸試算模組

本版本新增「全域房貸試算」能力。此模組的設計重點是把銀行核貸與買方現金壓力從物件價格中拆出來比較，但不把裝修預備金每坪費用放入房貸試算。裝修金額屬於使用者後續營運、設計與施工策略，不一定與物件坪數成正比，因此不應干擾買房條件本身的比較。

### 15.1 全域資料結構 `mortgageParams`

`mortgageParams` 是全物件共用參數，與 `weights`、`distanceTargets`、`constraints` 同屬於評估案層級設定，不應寫入單一物件 rawData。

```js
{
  loanToValuePercent: 70,
  annualInterestRatePercent: 2.3,
  loanTerms: [20, 30, 40],
  primaryTermYears: 30,
  gracePeriodYears: 0,
  maxLoanAmount: 0,
  transactionCostPercent: 3,
  monthlyPaymentLimit: 0
}
```

| key | 名稱 | 單位 | 說明 |
| --- | --- | --- | --- |
| `loanToValuePercent` | 貸款成數 | % | 例如 70 表示可貸基準價的 70%。 |
| `annualInterestRatePercent` | 年利率 | % | 用於本息平均攤還月付試算。 |
| `loanTerms` | 貸款年期 | 年 | 可輸入多個年期，例如 `20,30,40`，評比排行會同欄顯示。 |
| `primaryTermYears` | 壓力判斷年期 | 年 | 月付壓力主要用此年期檢核。 |
| `gracePeriodYears` | 寬限期 | 年 | 若大於 0，顯示寬限期僅繳息月付。 |
| `maxLoanAmount` | 最大貸款金額 | 萬 | 0 代表不限制；大於 0 時可貸金額不可超過此值。 |
| `transactionCostPercent` | 交易成本率 | % | 用於粗估稅費、代書、規費等一次性成本。 |
| `monthlyPaymentLimit` | 每月可負擔上限 | 元/月 | 0 代表不檢核月付壓力。 |

### 15.2 貸款基準價

房貸試算的貸款基準價固定等於該物件的賣家開價，也就是 `rawData.totalPrice`。此欄不得使用市場估值、實價登錄、手動核貸價或 CSV 匯入的衍生欄位。

原因是本系統的房貸模組用於比較買方在不同開價下的資金壓力；市場估值與實價登錄屬於行情分析，不應混入貸款基準，否則可貸金額、自備款與月付會被行情估值牽動，導致使用者誤判實際購買資金需求。

### 15.3 房貸公式

所有房價、可貸金額、自備款與初期現金欄位若標示「萬」即沿用物件價格單位；月付金額與每月可負擔上限一律以「元/月」呈現。因此貸款金額進入月付公式前必須先從「萬」換算成「元」。

```text
貸款基準價 = 賣家開價 = rawData.totalPrice
可貸金額 = min(貸款基準價 * 貸款成數, 最大貸款金額)
自備款 = max(0, 賣家開價 - 可貸金額)
交易成本 = 賣家開價 * 交易成本率
初期現金需求 = 自備款 + 交易成本
貸款缺口 = max(0, 賣家開價 - 貸款基準價)
```

在目前規格下，因貸款基準價固定等於賣家開價，`貸款缺口` 通常為 0。若未來要納入銀行鑑價折減，應新增獨立欄位與明確公式，而不是重用市場估值或 CSV 報表欄位。

本息平均攤還月付：

```text
月利率 = 年利率 / 12
期數 = 貸款年期 * 12
貸款金額(元) = 貸款金額(萬) * 10,000
月付(元/月) = 貸款金額(元) * 月利率 * (1 + 月利率)^期數 / ((1 + 月利率)^期數 - 1)
```

若年利率為 0：

```text
月付(元/月) = 貸款金額(元) / 期數
```

寬限期僅繳息月付：

```text
寬限期月付(元/月) = 貸款金額(萬) * 10,000 * 年利率 / 12
```

月付壓力判斷：

```text
若 monthlyPaymentLimit = 0：未檢核
若主要年期月付 > monthlyPaymentLimit：超標
若主要年期月付 > monthlyPaymentLimit * 0.9：接近上限
否則：可負擔
```

`貸款缺口` 的用途是提醒使用者：當賣家開價高於銀行較可能承認的核貸基準時，即使買方願意出價，也可能需要額外現金補足核貸不足。

1. 貸款基準價。
2. 可貸金額。
3. 自備款。
4. 初期現金需求。
5. 主要年期月付。
6. 月付壓力。
7. 補充文字顯示基準來源、貸款成數、年利率、貸款缺口與各年期月付。
8. 若有設定寬限期，補充文字與評比排行會顯示寬限期僅繳息月付。

評比排行頁新增「房貸試算」項目列：

| 項目 | 顯示方式 | 排序依據 |
| --- | --- | --- |
| 貸款基準 / 可貸金額 | 同一格上下顯示基準價、可貸金額、基準來源 | 可貸金額 |

### 15.5 CSV 匯入匯出規格

完整資料 CSV 新增獨立區段 `GLOBAL_MORTGAGE_PARAMS`，位置介於 `GLOBAL_CONSTRAINTS` 與 `PROPERTIES` 之間。使用者仍只需按一次匯出或匯入，系統會在同一份 CSV 中保存全域設定與物件資料。

```csv
#SECTION,GLOBAL_MORTGAGE_PARAMS
參數,值,單位
loanToValuePercent,70,%
annualInterestRatePercent,2.3,%
loanTerms,"20,30,40",年
primaryTermYears,30,年
gracePeriodYears,0,年
maxLoanAmount,0,萬
transactionCostPercent,3,%
monthlyPaymentLimit,0,元/月
```

匯入防呆：

1. 缺少 `GLOBAL_MORTGAGE_PARAMS` 時，保留目前畫面設定；若是載入雲端資料則使用預設值補齊。
2. 數值缺漏或非數字時使用 0 或既有預設值。
3. `loanTerms` 可接受逗號、空白、分號分隔，系統會去重並限制最多 5 個年期。
4. 舊版 CSV 若包含 `loanBaseMode` 或 `manualLoanBasePrice`，匯入時必須忽略；貸款基準價仍固定使用 `rawData.totalPrice`。
5. 房貸衍生欄位只在匯出時提供報表參考；匯入時必須忽略，不能覆寫計算結果。
6. 匯入 CSV 時必須先移除所有 derived 報表欄位，例如 `貸款基準價`、`可貸金額`、`自備款`、`交易成本`、`初期現金需求`、`主要年期月付`、`月付壓力`、`貸款缺口`。這些欄位永遠由目前 rawData 與全域房貸參數重新計算。
7. 載入雲端資料集時也必須清除物件上的 `derived`、`weightedScore`、`groups`、`constraintIssues` 等執行期欄位，只保留 `rawData`、`scores`、`distances` 與物件基本資料。

完整資料 CSV 的 `PROPERTIES` 區段會額外匯出下列衍生欄位：

| 欄位 | 說明 |
| --- | --- |
| `貸款基準價` | 依全域房貸參數推估的核貸基準。 |
| `可貸金額` | 貸款基準價乘以貸款成數後，套用最大貸款金額限制。 |
| `自備款` | 賣家開價扣除可貸金額。 |
| `交易成本` | 賣家開價乘以交易成本率。 |
| `初期現金需求` | 自備款加交易成本。 |
| `主要年期月付` | 依主要年期計算的本息平均攤還月付。 |
| `月付壓力` | 依每月可負擔上限判斷。 |
| `貸款缺口` | 賣家開價高於貸款基準價的差額。 |

重要原則：上述欄位是報表結果，不是資料來源。舊版 CSV 若含有錯誤的房貸衍生值，匯入後也不得沿用，系統必須以目前公式重新計算。

### 15.6 開發與驗證注意事項

1. 房貸參數是全域設定，必須納入 local state、CSV、雲端資料集 payload。
2. Firebase 未設定或未登入時，房貸試算仍必須可在本機正常使用。
3. 匯入舊版 CSV 不應報錯；缺少房貸區段時使用預設值。
4. 裝修預備金每坪費用不得放入房貸試算；現有的硬體扣除額只屬於市場估值與出價分析。
5. 建置測試需確認物件卡片、評比排行、完整 CSV、排行對比 CSV 均可正常產生。

### 15.7 評比排行錨定列水平同步

評比排行頁的錨定項目會以頁面級 sticky 區塊固定在畫面上方，但其欄位仍必須與主比較表的物件欄位保持水平對齊。因為主表格與錨定摘要各自有橫向捲動容器，實作時需同步兩者的 `scrollLeft`：

1. 主比較表向左或向右拖拉時，錨定摘要同步移動到相同水平位置。
2. 錨定摘要本身若被橫向拖拉，主比較表也同步到相同水平位置。
3. 錨定項目新增、取消或物件排序改變後，錨定摘要需重新套用主比較表目前的 `scrollLeft`。
4. 同步時需避免兩個 `onScroll` 互相觸發造成抖動，可用同步旗標或 `requestAnimationFrame` 控制。
5. 驗證時應在多物件、寬表格、頁面垂直捲動到中段後測試，確認錨定列與主表格的物件欄位仍上下對齊。
6. 錨定摘要與主比較表必須共用相同的欄寬設定。建議兩張表都使用相同的 `colgroup`，第一欄固定為項目欄寬，後續物件欄固定為同一物件欄寬；不可只依 `min-width` 與內容自動撐開，否則照片、地址或較長文字會讓上下欄位錯位。
7. 錨定摘要中的每個項目列都應提供「移除錨定」按鈕，使用者不需要捲回原本項目列才能取消錨定。
8. 錨定摘要應採精簡表頭。主比較表可以顯示照片縮圖協助辨識物件，但錨定摘要固定在畫面上方，若同步顯示照片會佔用過多垂直空間，因此錨定摘要表頭以欄序與物件名稱為主，不顯示照片縮圖。

### 15.8 評比排行互動編輯

評比排行頁除了展示比較結果，也應支援使用者在比較情境下快速調整資料：

1. 項目顯示控制需同時提供「全部顯示」與「全部取消」。全部取消會暫時隱藏所有項目列，使用者可再逐一勾選需要比較的項目。
2. 物件欄位需支援拖拉排序。系統預設仍可依總分、總價、坪數、距離等數值項目排序；但使用者可手動把任一物件拖到相鄰欄位，方便把非連續名次的物件放在一起比較。
3. 使用者點選任何數值項目的排序按鈕時，系統會清除手動物件欄順序並回到該項排序結果；使用者再次拖拉物件欄時，則以手動順序為準。
4. 評分列在評比排行頁需可直接輸入 0-10 分。使用者通常是在橫向比較所有物件時最容易判斷主觀分數，因此不應強迫使用者回到物件卡片逐一調整。
5. 評比排行直接修改分數後，系統需即時計算加權總分、排行榜、錨定摘要與匯出報表。此行為應與物件管理卡片中的評分滑桿共用同一份 `properties[].scores` 狀態。

### 15.9 評比排行項目分群

評比排行項目不應只用單層清單呈現。當比較項目增加後，使用者常會依目的切換一整組資料，例如只看貸款壓力、只看坪數空間、只看價格行情。因此項目顯示控制需採分群設計：

1. 每個大分群可一次顯示或隱藏整組項目。
2. 分群內仍需保留每一個項目的獨立勾選。
3. 分群卡片需用不同色條或淡色背景區分，讓使用者能快速掃描。
4. 分群數量不宜過多，建議維持 6 到 7 群；若群太多，會失去快速切換目的。

建議分群如下：

| 分群 | 用途 | 主要項目 |
| --- | --- | --- |
| 決策總覽 | 先快速判斷候選物件是否值得深入看 | 加權總分、不符條件、市場估值、建議下錨價、最高追價、開價偏離 |
| 價格行情 | 判斷賣方開價與市場是否合理 | 總價、實價登錄、近一年均價/中位數、歷史高低均價、各種單價差異 |
| 貸款現金流 | 評估買方資金壓力與銀行核貸風險 | 貸款基準、可貸金額、自備款、初期現金、月付、貸款缺口 |
| 坪數空間 | 比較實際可用面積與空間結構 | 建坪、1F 面積、附屬面積、夾層、公設、土地、車位坪數 |
| 位置交通 | 比較外部環境與到特定目的地的距離 | 地址、行政區、商圈、臨路、停車環境、全域距離目標 |
| 格局硬體 | 比較建物條件、展示面與硬體改善成本 | 面寬、深度、路寬、方正、挑高、採光、角間、屋齡、硬體備註 |
| 策略評分 | 直接調整主觀評分並觀察排名變化 | 9 項策略維度 0-10 分 |

分群規則應以 row id 判斷，例如 `price.*` 與 `market.*` 放入價格行情、`mortgage.*` 放入貸款現金流、`distance.*` 放入位置交通、`score.*` 放入策略評分。`raw.*` 欄位則依欄位 key 再細分到坪數空間、位置交通或格局硬體。

### 15.10 GitHub Pages 與 Firebase 部署設定

本系統可透過 GitHub Actions 部署到 GitHub Pages。部署時需注意 Firebase 設定：

1. GitHub Actions 的 build 階段必須提供 `VITE_FIREBASE_API_KEY`、`VITE_FIREBASE_AUTH_DOMAIN`、`VITE_FIREBASE_PROJECT_ID`、`VITE_FIREBASE_STORAGE_BUCKET`、`VITE_FIREBASE_MESSAGING_SENDER_ID`、`VITE_FIREBASE_APP_ID`。
2. 這些 Firebase Web config 會被打包進前端程式，屬於瀏覽器端公開設定，不應依賴它作為資料安全邊界。
3. 資料隔離必須依靠 Firebase Authentication 與 Firestore Security Rules，建議規則限制使用者只能讀寫 `users/{uid}/datasets/{datasetId}` 中自己的 `uid`。
4. 若使用 Google 登入，Firebase Console > Authentication > Settings > Authorized domains 必須加入實際使用網域。GitHub Pages 版本需加入 `phonm.github.io`；本機開發需加入 `localhost` 與 `127.0.0.1`。若沒有加入，登入或註冊流程會出現授權網域錯誤。
5. 若想避免把專案預設 Firebase config 寫在 workflow，可改用 GitHub Repository Variables 設定同名 `VITE_FIREBASE_*` 變數；workflow 會優先使用 GitHub Variables。
6. 本機 `.env` 不會提交到 GitHub，僅供本機開發使用。
